Forensic — Blockchain Forensic

Transaction Forensic & Smart Contract Recovery

Blockchain forensic memanfaatkan sifat blockchain yang transparan dan immutable — semua transaksi tersimpan permanen dan bisa dianalisis. Tujuannya: melacak aliran dana mencurigakan, mengidentifikasi pelaku, dan memulihkan data dari smart contract.

Fondasi — Mengapa blockchain bisa dianalisis forensik

Blockchain = ledger publik yang permanen dan transparan

Berbeda dari rekening bank yang privat, semua transaksi blockchain tercatat di ledger publik yang bisa dibaca siapapun. Ini membuat blockchain LEBIH mudah dilacak daripada tunai!

Mitos: "Bitcoin anonim"

Salah! Bitcoin pseudonim — alamat terlihat, tapi identitas asli tersembunyi sampai ada link ke dunia nyata.

Fakta: Blockchain forensics powerful

Chainalysis, Elliptic, TRM Labs bisa melacak aliran dana ransomware, exchange hack, dan money laundering.

Properti blockchain yang dimanfaatkan forensik:

Immutability: transaksi tidak bisa dihapus → jejak permanen
Transparency: semua tx publik → bisa ditelusuri siapapun
Pseudonymity: alamat terlihat, identitas perlu di-link ke off-chain data
Graph structure: tx membentuk graph → bisa ditelusuri dengan graph analysis

Langkah 4 — Identifikasi lead investigasi terkuat

1ExchangeKYC adalah titik kritis investigasi

Dari semua alamat yang teridentifikasi, 1ExchangeKYC adalah yang paling penting:

Mengapa exchange adalah lead terkuat?
Exchange dengan KYC (Know Your Customer) menyimpan data identitas user. Ini bisa di-subpoena oleh penegak hukum untuk mendapatkan nama asli, email, nomor telepon, bahkan alamat fisik pelaku.
Prosedur selanjutnya:
1. Kirim legal request (subpoena) ke exchange
2. Dapatkan KYC data dari akun yang menerima 1Out222 dan 1Out333
3. Kalau exchange di yurisdiksi berbeda → gunakan MLA (Mutual Legal Assistance)
Bahkan jika pelaku pakai mixer, kalau mereka punya satu titik kontak dengan exchange KYC, identitas mereka bisa terungkap. Ini kelemahan utama crypto criminals!
Dua aliran lain (1Out111, 1Out444) masih belum jelas. Perlu dilanjutkan dengan transaction tracing di blok berikutnya.
Langkah 5 — Heuristic: Change detection

Identifikasi "kembalian" untuk temukan wallet yang sama

Heuristic paling berguna: dalam sebuah transaksi Bitcoin, sering ada satu output yang merupakan "kembalian" ke pengirim. Mendeteksi ini membantu cluster alamat.

Indikator kembalian (change output): 1. Nilai yang tidak bulat (round number = pembayaran, tidak bulat = kembalian) 2. Alamat baru yang tidak pernah muncul sebelumnya 3. Output yang nilainya lebih kecil dari input 4. Alamat yang sama dengan input (reuse)

Analisis TX-B:

Output 1: 1MixerAAA = 1.95 BTC (pembayaran ke mixer)
Output 2: 1RansomXXX = 0.04 BTC ← KEMBALIAN! (alamat sama dengan input)
Ketahuan! 1RansomXXX masih memegang 0.04 BTC. Kalau pelaku bertransaksi lagi dari sini, bisa dilacak lebih lanjut.

Langkah 1 — Heuristic 1: Common Input Ownership

Input dalam satu transaksi = satu pemilik

Untuk membayar Bitcoin, kamu harus sign semua input dengan private key kamu. Jadi kalau satu transaksi punya banyak input, semua input itu dimiliki satu entitas (yang punya semua private key-nya).

Aturan: jika AddrX dan AddrY sama-sama jadi input di satu TX, maka AddrX dan AddrY dimiliki entitas yang SAMA.

Terapkan ke TX-1:

TX-1 inputs: [AddrA, AddrB] → AddrA dan AddrB dimiliki entitas yang SAMA!

Terapkan ke TX-2:

TX-2 inputs: [AddrC, AddrD] → AddrC dan AddrD dimiliki entitas yang SAMA!
Dari TX-1: cluster {AddrA, AddrB}. Dari TX-2: cluster {AddrC, AddrD}. Tapi belum tahu apakah kedua cluster ini milik orang yang sama!

Langkah 3 — Gabungkan heuristic: merge cluster

Trace ownership chain dari TX ke TX

Sekarang kita bisa gabungkan semua informasi:

TX-1: inputs={AddrA,AddrB} → Common input ownership → {AddrA,AddrB} adalah satu cluster
TX-1: change=AddrC → AddrC milik pengirim TX-1 = cluster {AddrA,AddrB} → merge: {AddrA,AddrB,AddrC}
TX-2: inputs={AddrC,AddrD} → AddrC sudah di cluster → AddrD join cluster: {AddrA,AddrB,AddrC,AddrD}
TX-2: change=AddrE → AddrE milik pengirim TX-2 = cluster → merge: {AddrA,AddrB,AddrC,AddrD,AddrE}
TX-3: input={AddrE} → AddrE sudah di cluster → TX-3 dikirim oleh cluster yang sama
TX-3: change=AddrF → AddrF milik pengirim TX-3 → merge: {AddrA,AddrB,AddrC,AddrD,AddrE,AddrF}
Cluster final: {AddrA, AddrB, AddrC, AddrD, AddrE, AddrF} Semua 6 alamat ini dimiliki oleh SATU entitas!

Langkah 1 — Identifikasi jenis exploit: Reentrancy Attack

Reentrancy = memanggil kembali fungsi sebelum state di-update

Dari pola internal calls: withdraw → transfer ETH → (receive) → withdraw lagi → ini adalah reentrancy attack — serangan paling terkenal di Ethereum (DAO Hack 2016: $60M).

// Kode VictimDeFi yang VULNERABLE: mapping(address => uint) public balances; function withdraw(uint amount) public { require(balances[msg.sender] >= amount); // 1. Transfer dulu ← BAHAYA! external call sebelum state update (bool sent,) = msg.sender.call{value: amount}(""); require(sent); // 2. Update balance SETELAH transfer ← terlambat! balances[msg.sender] -= amount; // ← attacker sudah re-enter sebelum ini! }
CEI (Checks-Effects-Interactions) pattern harus diikuti: dulu Check (require), lalu Effects (update state), baru Interactions (external call). Kode di atas melanggar ini!
Langkah 2 — Trace seluruh internal transaction

Decode call stack dari TX exploit

Di Ethereum, satu transaksi bisa menghasilkan ratusan internal transactions. Tools: Tenderly, Etherscan Debug, Phalcon (Blocksec).

CALL attack() from 0xAttacker to 0xAttackContract
CALL withdraw(1 ETH) to 0xVictimDeFi [balance[attacker]=1ETH ✓]
CALL value=1ETH to 0xAttackContract (transfer)
CALL withdraw(1ETH) to 0xVictimDeFi [balance masih 1ETH!] ← REENTRY
CALL value=1ETH (lagi!) ... loop 10x total
balance[attacker] -= 1ETH (update, tapi sudah terlambat!)
10 × withdraw(1ETH) sukses meski deposit hanya 1 ETH → 10 ETH keluar dari contract!

Langkah 5 — Smart Contract Recovery: Decode storage

Baca storage contract untuk recovery data

Smart contract menyimpan data di "storage slots". Kita bisa baca semua storage bahkan dari contract tanpa source code!

## Baca storage contract di block SEBELUM exploit untuk forensic: from web3 import Web3 w3 = Web3(Web3.HTTPProvider('https://mainnet.infura.io/v3/KEY')) contract = "0xVictimDeFi" block_before = 18499999 # block sebelum exploit # Baca mapping balances (slot 0) # Storage slot untuk mapping[key] = keccak256(abi.encode(key, slot_number)) import eth_abi from eth_hash.auto import keccak attacker = "0xAttacker" slot_key = keccak(eth_abi.encode(['address', 'uint256'], [attacker, 0])) balance_at_slot = w3.eth.get_storage_at( contract, slot_key, block_identifier=block_before) print(f"Balance attacker sebelum exploit: {int(balance_at_slot.hex(), 16) / 1e18} ETH") # Output: Balance attacker sebelum exploit: 1.0 ETH # Baca total ETH di contract: balance = w3.eth.get_balance(contract, block_identifier=block_before) print(f"Total ETH sebelum exploit: {balance / 1e18} ETH") # Output: Total ETH sebelum exploit: 50.0 ETH
Storage forensic berguna untuk: membuktikan state sebelum exploit, hitung kerugian tepat, dan bisa jadi bukti di pengadilan.
Langkah 6 — Analisis bytecode attack contract

Reverse engineer attack contract untuk pahami metode exploit

## Dapatkan bytecode attack contract: bytecode = w3.eth.get_code("0xAttackContract") ## Decompile dengan Panoramix atau Heimdall: $ heimdall decompile 0xAttackContract --rpc-url https://mainnet.infura.io/... ## Hasil decompile (simplified): function attack() payable { victim.deposit{value: 1 ether}(); victim.withdraw(1 ether); } receive() external payable { if (address(victim).balance >= 1 ether) { victim.withdraw(1 ether); // ← reentrancy! } } ## Signature fungsi (4 byte selector): attack() = 0x9e5faafc withdraw(uint256) = 0x2e1a7d4d ← terlihat di calldata TX exploit
Bahkan tanpa source code, bytecode bisa di-decompile untuk mendapatkan logic exploit. Ini penting untuk: memahami vektor serangan, menulis patch, dan laporan forensik.
