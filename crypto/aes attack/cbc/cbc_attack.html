CBC Attack

Dua serangan utama pada AES-CBC

CBC punya dua kelemahan berbeda yang bisa diserang dengan cara berbeda. Pilih serangan yang ingin dipelajari:

Soal — CBC Bit-Flipping Attack

Ubah "user" menjadi "admi" tanpa kunci!

Sebuah web app mengenkripsi cookie sesi dengan AES-CBC (blok 16 byte). Cookie plaintext-nya adalah:

Blok 1 (byte 0-15): userid=12345678
Blok 2 (byte 16-31): role=user&xxx=0

Kamu berhasil mencuri ciphertext cookie dan IV-nya. Ubah "user" di blok 2 menjadi "admi" (untuk akses admin) dengan memodifikasi ciphertext — tanpa tahu kunci enkripsi!

IV =
00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00
C₁ (enkripsi blok 1) =
3F A2 9C 01 E7 48 BB F3 2D 77 5A C9 11 6E D4 82
C₂ (enkripsi blok 2) =
8B 44 1D 76 C2 93 5E 0A F1 20 6E B9 3A D5 47 CC

Langkah 1 — Pahami rumus dekripsi CBC

Kunci dari serangan ini ada di rumus dekripsi CBC

Saat server mendekripsi ciphertext, rumusnya adalah:

P_i = AES_K⁻¹(C_i) ⊕ C_{i-1}

Untuk blok pertama (i=1), C₀ = IV. Jadi:

P₂ = AES_K⁻¹(C₂) ⊕ C₁

Sekarang, kalau kita ganti C₁ dengan C₁' yang sudah dimodifikasi:

P₂' = AES_K⁻¹(C₂) ⊕ C₁'
AES_K⁻¹(C₂) tidak berubah — karena C₂ tidak kita ubah! Yang berubah hanya C₁. Ini yang akan kita eksploitasi.
Trade-off: blok C₁ yang kita modifikasi akan mendekripsi menjadi sampah acak di P₁. Tapi di serangan ini kita tidak peduli isi P₁ — kita hanya peduli P₂!
Langkah 2 — Identifikasi target byte

Di posisi byte berapa kata "user" berada?

Blok 2 plaintext: role=user&xxx=0

Posisi dalam blok 20123456789101112131415
Karakterrole=user&xxx=0—
Byte index (global)16171819202122232425262728293031
"user" ada di posisi 5,6,7,8 dalam blok 2 — atau byte ke-21,22,23,24 secara global.

Karena P₂ dipengaruhi oleh C₁, kita harus modifikasi byte ke-5,6,7,8 dari C₁ (0-indexed) untuk mengubah byte yang sama di P₂.

Langkah 3 — Turunkan rumus modifikasi

Hitung nilai C₁' yang diperlukan

Kita ingin P₂'[j] = target_char. Dari rumus dekripsi:

P₂[j] = AES_K⁻¹(C₂)[j] ⊕ C₁[j] P₂'[j] = AES_K⁻¹(C₂)[j] ⊕ C₁'[j]

XOR kedua persamaan:

P₂[j] ⊕ P₂'[j] = C₁[j] ⊕ C₁'[j]

Isolasi C₁'[j]:

C₁'[j] = C₁[j] ⊕ P₂[j] ⊕ P₂'[j]
Formula ini adalah inti dari seluruh serangan. Kita perlu tahu: nilai C₁[j] asli (punya), nilai P₂[j] asli (punya = plaintext kita ketahui), dan nilai P₂'[j] yang kita inginkan.

Dengan kata lain: C₁'[j] = C₁[j] XOR original_char XOR target_char

Langkah 4 — Hitung modifikasi untuk 'u' → 'a'

Byte ke-5 blok 2: 'u' (0x75) → 'a' (0x61)

Data yang kita punya:

VariabelNilaiKeterangan
C₁[5]0x93Byte ke-5 dari C₁ asli
P₂[5] = 'u'0x75Plaintext asli yang kita ketahui
P₂'[5] = 'a'0x61Target yang kita inginkan
C₁'[5] = C₁[5] ⊕ P₂[5] ⊕ P₂'[5] = 0x93 ⊕ 0x75 ⊕ 0x61

Hitung step by step:

0x93 ⊕ 0x75: 1001 0011 ⊕ 0111 0101 = 1110 0110 = 0xE6 0xE6 ⊕ 0x61: 1110 0110 ⊕ 0110 0001 = 1000 0111 = 0x87 C₁'[5] = 0x87

Ganti byte ke-5 C₁ dari 0x93 → 0x87. Lanjut ke byte berikutnya.

Langkah 5 — Hitung modifikasi untuk 's','e','r' → 'd','m','i'

Tiga byte sisanya: byte 6,7,8

Byte ke-6: 's' (0x73) → 'd' (0x64)

C₁'[6] = C₁[6] ⊕ 's' ⊕ 'd' = 0x5E ⊕ 0x73 ⊕ 0x64 = 0x5E ⊕ 0x17 (0x73⊕0x64 = 0x17) = 0x49

Byte ke-7: 'e' (0x65) → 'm' (0x6D)

C₁'[7] = C₁[7] ⊕ 'e' ⊕ 'm' = 0x0A ⊕ 0x65 ⊕ 0x6D = 0x0A ⊕ 0x08 (0x65⊕0x6D = 0x08) = 0x02

Byte ke-8: 'r' (0x72) → 'i' (0x69)

C₁'[8] = C₁[8] ⊕ 'r' ⊕ 'i' = 0xF1 ⊕ 0x72 ⊕ 0x69 = 0xF1 ⊕ 0x1B (0x72⊕0x69 = 0x1B) = 0xEA
Semua 4 byte modifikasi sudah dihitung: C₁'[5]=0x87, C₁'[6]=0x49, C₁'[7]=0x02, C₁'[8]=0xEA
Langkah 6 — Rakit C' yang sudah dimodifikasi

Ganti byte di C₁, kirim (IV, C₁', C₂) ke server

C₁ asli vs C₁' yang sudah dimodifikasi:

Byte012345678...

Ingat: P₁ akan rusak total (sampah acak). Tapi server tetap menerima kalau dia tidak validasi integritas P₁ — dia hanya parse role= dari P₂.
Langkah 7 — Verifikasi hasil dekripsi

Apa yang server dekripsi dari C₁' dan C₂?

P₂' = AES_K⁻¹(C₂) ⊕ C₁' Karena AES_K⁻¹(C₂) tidak berubah, dan kita sudah hitung: C₁'[5] dipilih agar P₂'[5] = 'a' ✓ C₁'[6] dipilih agar P₂'[6] = 'd' ✓ C₁'[7] dipilih agar P₂'[7] = 'm' ✓ C₁'[8] dipilih agar P₂'[8] = 'i' ✓ P₂' = "role=admi&xxx=0"
Hasil serangan
role=admi → akses admin berhasil!

Ringkasan Bit-flipping Attack

1. Identifikasi posisi byte target di plaintext blok ke-i

2. Byte itu dikontrol oleh C_{i-1} di posisi yang sama

3. Hitung: C'[j] = C[j] ⊕ original_byte ⊕ target_byte

4. Kirim ciphertext yang sudah dimodifikasi ke server

5. Blok C_{i-1} sendiri akan rusak saat didekripsi — tidak masalah

Syarat: attacker tahu plaintext aslinya (atau bisa menduga strukturnya)

Soal — Padding Oracle Attack

Dekripsi ciphertext tanpa kunci menggunakan error server!

Server memiliki kelemahan: ia mengembalikan pesan error berbeda untuk "padding salah" vs "padding benar tapi MAC gagal". Ini sudah cukup untuk mendekripsi seluruh ciphertext!

Ciphertext yang ingin didekripsi (2 blok, 16 byte/blok):
C₁ = 3F A2 9C 01 E7 48 BB F3 2D 77 5A C9 11 6E D4 82
C₂ = 8B 44 1D 76 C2 93 5E 0A F1 20 6E B9 3A D5 47 CC

Server response: {"error":"invalid padding"} atau {"error":"invalid MAC"}

Langkah 1 — Pahami PKCS#7 Padding

Apa itu padding dan kenapa penting?

AES bekerja dengan blok 16 byte. Kalau plaintext tidak pas 16 byte, ditambah padding. Standar yang dipakai: PKCS#7.

Aturan PKCS#7: tambahkan N byte, tiap byte bernilai N
Plaintext 13 byte → butuh 3 byte padding → tambah 03 03 03
Plaintext 15 byte → butuh 1 byte padding → tambah 01
Plaintext 16 byte → butuh 16 byte padding → tambah 10 10 10 10 10 10 10 10 10 10 10 10 10 10 10 10
Server selalu cek padding setelah dekripsi. Kalau padding tidak valid (misal byte terakhir adalah 0x03 tapi dua sebelumnya bukan 0x03), server lempar error "invalid padding".
Inilah oracle-nya! Perbedaan respons "invalid padding" vs "invalid MAC" membocorkan informasi tentang byte terakhir plaintext yang terdekripsi.
Langkah 2 — Inti matematika Padding Oracle

Eksploitasi rumus dekripsi CBC

Ingat rumus dekripsi CBC:

P₂[j] = AES_K⁻¹(C₂)[j] ⊕ C₁[j]

Misalkan kita tahu AES_K⁻¹(C₂)[15] = nilai misterius yang kita sebut I₂[15] (intermediate). Maka:

P₂[15] = I₂[15] ⊕ C₁[15]

Kalau kita kirim C₁' yang byte terakhirnya kita ubah menjadi nilai X:

P₂'[15] = I₂[15] ⊕ X

Untuk padding valid dengan nilai 0x01, kita butuh P₂'[15] = 0x01:

I₂[15] ⊕ X = 0x01 → X = I₂[15] ⊕ 0x01
Jadi: coba semua 256 nilai X (0x00 sampai 0xFF) sampai server bilang padding valid. Kalau valid, kita tahu I₂[15] = X ⊕ 0x01. Lalu P₂[15] = I₂[15] ⊕ C₁[15] asli!
Langkah 3 — Ekstrak byte terakhir P₂

Brute-force byte ke-15 (byte terakhir) C₁

C₁[15] asli = 0x82. Kita coba semua nilai X dari 0x00 sampai 0xFF untuk C₁'[15] dan kirim ke server:

C₁'[15] dicobaServer responseKeterangan
0x00invalid paddingP₂'[15] = I₂[15]⊕0x00 bukan 0x01
0x01invalid padding...
.........
0x83invalid MAC ✓padding valid! P₂'[15] = 0x01
...invalid padding...

Server bilang valid saat X = 0x83. Artinya:

I₂[15] ⊕ 0x83 = 0x01 I₂[15] = 0x01 ⊕ 0x83 = 0x82 P₂[15] = I₂[15] ⊕ C₁[15] = 0x82 ⊕ 0x82 = 0x00
Byte terakhir P₂ = 0x00! (kemungkinan padding atau null byte). Dengan cara yang sama untuk semua 16 byte, seluruh P₂ bisa dikupas.
Langkah 4 — Lanjut ke byte ke-14

Untuk byte ke-14, target padding = 0x02 0x02

Kita sudah tahu I₂[15] = 0x82. Sekarang ekstrak I₂[14]. Untuk padding valid 0x02, kita butuh P₂'[15] = 0x02 DAN P₂'[14] = 0x02.

Fix dulu byte ke-15 agar menghasilkan 0x02:

Ingin P₂'[15] = 0x02 C₁'[15] = I₂[15] ⊕ 0x02 = 0x82 ⊕ 0x02 = 0x80 ← fix di sini

Lalu brute-force C₁'[14] (0x00–0xFF) sampai padding valid:

Kalau valid saat C₁'[14] = Y: I₂[14] ⊕ Y = 0x02 I₂[14] = Y ⊕ 0x02 P₂[14] = I₂[14] ⊕ C₁[14]
Pola ini berulang untuk tiap byte: untuk byte ke-j, target padding = (16-j), dan semua byte setelahnya sudah difix menggunakan nilai I yang sudah diketahui.

Langkah 6 — Edge case: 2 nilai valid

Hati-hati: kadang ada 2 X yang menghasilkan padding valid!

Saat mencari byte terakhir, bisa ada 2 nilai yang menghasilkan "valid":

Kasus benar (0x01)

P₂'[15] = 0x01 → valid karena padding 1 byte

Kebetulan valid

P₂'[15] = 0x02 dan P₂'[14] kebetulan = 0x02 juga → valid!

Cara bedain: kalau dapat 2 nilai valid, modifikasi juga byte ke-14 (set ke nilai random). Kalau masih valid → itu yang 0x01. Kalau jadi invalid → yang tadi kebetulan.

Di implementasi nyata (tools seperti padbuster, python-paddingoracle), edge case ini sudah ditangani otomatis.
Langkah 7 — Padding Oracle untuk enkripsi (bonus)

Padding Oracle juga bisa ENKRIPSI arbitrary plaintext!

Bukan hanya dekripsi — Padding Oracle bisa membuat ciphertext valid dari plaintext apapun, tanpa kunci!

Ingin enkripsi P' = "admin\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b" (dengan padding PKCS#7 yang valid)

Caranya:

1. Mulai dari blok terakhir: set C' = blok acak
2. Gunakan Padding Oracle untuk cari C'_{i-1} yang menghasilkan plaintext yang kita inginkan
3. Ingat: P'[j] = I[j] ⊕ C'[j] → C'[j] = I[j] ⊕ P'[j]
4. Dan I[j] sudah kita tahu dari proses oracle sebelumnya!
5. Ulangi untuk tiap blok dari belakang ke depan
Ini sangat berbahaya: attacker bisa forge ciphertext valid tanpa kunci. Inilah kenapa CBC WAJIB dikombinasikan dengan MAC (misal HMAC) untuk cegah tampering.

Ringkasan Padding Oracle Attack

1. Temukan oracle: server mengembalikan error berbeda untuk padding valid vs invalid

2. Untuk setiap blok Cᵢ, brute-force byte C_{i-1} dari belakang ke depan

3. Saat padding valid → dapatkan nilai I (intermediate) = AES_K⁻¹(Cᵢ)

4. Hitung plaintext: Pᵢ[j] = I[j] ⊕ C_{i-1}[j] asli

5. Ulangi untuk setiap byte dan setiap blok
