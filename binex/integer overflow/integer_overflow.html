Binary Exploitation — Integer Overflow / Underflow

Ketika bilangan "meluap" dan program menjadi tidak terduga

Integer Overflow/Underflow terjadi ketika operasi aritmetika menghasilkan nilai di luar batas tipe data. Komputer tidak error — ia diam-diam "wrap around" ke nilai yang tidak kita harapkan. Inilah yang dieksploitasi.

Fondasi — Tipe data integer di C/C++

Setiap tipe punya batas — dan batas itu bisa dilampaui

Di C/C++, integer disimpan dalam jumlah bit yang tetap. Kalau nilai melewati batas, ia wrap around (melingkar balik) — bukan error, tidak ada exception, program jalan terus dengan nilai yang salah.

Rumus Wrap Around

Cara hitung nilai setelah overflow/underflow

Untuk tipe N-bit unsigned: nilai_baru = (nilai_lama ± operasi) mod 2^N uint8_t: 255 + 1 = (255 + 1) mod 256 = 0 uint8_t: 0 - 1 = (0 - 1) mod 256 = 255 uint16_t: 65535 + 1 = (65535+1) mod 65536 = 0 uint32_t: 0 - 1 = 4294967295 Untuk tipe N-bit signed (two's complement): int8_t: 127 + 1 = -128 (MIN_INT) int8_t: -128 - 1 = 127 (MAX_INT) int32_t: 2147483647 + 1 = -2147483648
Kenapa berbahaya? Karena program melanjutkan eksekusi dengan nilai yang salah ini — dan programmer biasanya tidak mengeceknya. Inilah celah yang dieksploitasi.
Soal 1 — Integer Overflow Attack

Bypass pemeriksaan saldo dengan overflow!

Sebuah aplikasi belanja online memiliki fungsi berikut untuk memproses pembelian. Temukan cara untuk membeli item seharga 200 dengan saldo hanya 10!

// Tipe data: uint8_t (0 - 255)
uint8_t saldo = 10; // saldo user
uint8_t harga = 200; // harga item
uint8_t diskon = 0; // diskon dari voucher

// Harga setelah diskon
uint8_t harga_bayar = harga - diskon;

// Cek apakah saldo cukup
if (saldo >= harga_bayar) {
  saldo -= harga_bayar;
  printf("Pembelian berhasil!\n");
} else {
  printf("Saldo tidak cukup.\n");
}

Pertanyaan: Berikan nilai diskon berapa yang bisa membuat saldo 10 bisa membeli item 200?

1 / 6
Langkah 1 — Identifikasi celah

Di mana titik lemah kode ini?

Perhatikan baris ini:

uint8_t harga_bayar = harga - diskon;

Kalau diskon > harga, maka harga - diskon akan menghasilkan nilai negatif. Tapi karena tipenya uint8_t (unsigned, tidak bisa negatif), hasilnya akan wrap around ke nilai besar!

Harapan programmer

diskon selalu <= harga, jadi harga_bayar selalu positif dan masuk akal.

Realita yang bisa dieksploitasi

Kalau kita kontrol nilai diskon dan set > 200, harga_bayar akan underflow ke nilai kecil!

Ingat: uint8_t tidak bisa menyimpan nilai negatif. Hasil negatif akan wrap ke (256 + nilai_negatif).
Langkah 2 — Pahami wrap around untuk pengurangan

Apa yang terjadi saat 200 - diskon di uint8_t?

Rumus: untuk uint8_t, hasil = (200 - diskon) mod 256

Kita perlu harga_bayar <= 10 (saldo kita). Mari cari nilai diskon yang menghasilkan ini.

Langkah 3 — Hitung nilai diskon yang tepat

Cari diskon sehingga harga_bayar = 0 atau sangat kecil

Kita ingin harga_bayar = 0, artinya:

(200 - diskon) mod 256 = 0 200 - diskon = -256 (karena -256 mod 256 = 0) diskon = 200 + 256 = 456

Tapi diskon bertipe uint8_t juga! Jadi max 255. Kalau diskon = 200:

(200 - 200) mod 256 = 0 ✓
diskon = 200 → harga_bayar = 0! Saldo 10 >= 0, pembelian berhasil!

Atau kalau diskon bisa lebih dari 200 tapi masih uint8_t (max 255):

diskon = 205: (200 - 205) mod 256 = (-5) mod 256 = 251 → gagal diskon = 210: (200 - 210) mod 256 = (-10) mod 256 = 246 → gagal
Yang paling bersih adalah diskon = 200 → harga_bayar = 0. Atau diskon tepat = 200 + N dimana N = saldo yang tersisa.
Langkah 4 — Trace eksekusi program dengan diskon=200

Ikuti eksekusi kode step by step

uint8_t saldo = 10;
uint8_t harga = 200;
uint8_t diskon = 200; // ← input kita

uint8_t harga_bayar = 200 - 200;
// = 0 (tidak wrap, kebetulan 0)
// harga_bayar = 0

if (10 >= 0) { // TRUE!
  saldo -= 0; // saldo tetap 10
  printf("Pembelian berhasil!\n");
}
Dengan diskon=200, kita membeli item 200 secara GRATIS dan saldo tidak berkurang!
Langkah 5 — Skenario lebih realistis dengan wrap

Pakai diskon = 201: harga_bayar wrap ke 255

uint8_t harga_bayar = 200 - 201;
// Matematis: -1
// uint8_t: (-1) mod 256 = 255
// harga_bayar = 255 ← WRAP AROUND!

if (10 >= 255) { // FALSE
  // Tidak berhasil
}

Sebaliknya, coba diskon = 246 agar harga_bayar = 10:

(200 - 246) mod 256 = (-46) mod 256 = 210 → tidak = 10

Cari dengan formula: diskon = 200 - target_harga_bayar

Mau harga_bayar = 5: diskon = 200 - 5 = 195 ✓ Mau harga_bayar = 10: diskon = 200 - 10 = 190 ✓ Mau harga_bayar = 0: diskon = 200 - 0 = 200 ✓
Formula sederhana: diskon = harga - harga_yang_kita_mau_bayar
Langkah 6 — Cara fix dan ringkasan

Bagaimana programmer seharusnya fix ini?

// FIX 1: Validasi diskon tidak melebihi harga
if (diskon > harga) {
  printf("Diskon tidak valid!\n");
  return;
}

// FIX 2: Gunakan tipe yang lebih besar dan cek
int32_t harga_bayar = (int32_t)harga - (int32_t)diskon;
if (harga_bayar < 0) {
  printf("Diskon tidak valid!\n");
  return;
}
Jawaban soal
diskon = 200 → harga_bayar = 0
Beli item 200 dengan saldo 10, gratis, saldo tidak berkurang

Ringkasan Integer Overflow/Underflow Attack

1. Identifikasi operasi aritmetika pada tipe integer terbatas

2. Cek apakah hasilnya bisa negatif (pada unsigned) atau melewati MAX

3. Hitung nilai input yang menyebabkan wrap ke nilai yang menguntungkan

4. Rumus: hasil = (a ± b) mod 2^N untuk tipe N-bit

Soal 2 — Integer Underflow Attack

Bypass batas usia dengan underflow!

Sebuah sistem registrasi membatasi umur user minimum 18 tahun. Lihat kode ini dan temukan cara bypass!

uint8_t umur_input; // input dari user (0-255)
uint8_t batas_min = 18; // usia minimum
uint8_t selisih;

scanf("%hhu", &umur_input); // baca input

selisih = umur_input - batas_min;

if (selisih <= 100) { // "usia 18-118 diterima"
  printf("Registrasi berhasil!\n");
} else {
  printf("Usia tidak valid.\n");
}

Pertanyaan: Input umur berapa yang bisa lolos padahal < 18?

Bagaimana kalau input sangat besar tapi uint8_t?

Karena scanf("%hhu", ...) membaca uint8_t, input sudah dibatasi 0–255. Tapi mari periksa umur 119–255:

Hmm, semua aman. Tapi perhatikan: apakah scanf("%hhu") bisa menerima input lebih dari 255?

Kalau input "300" diberikan ke scanf dengan %hhu, hasilnya implementation-defined — tapi seringkali: 300 mod 256 = 44. Dan 44 - 18 = 26 <= 100 → LOLOS!
Input 300 → tersimpan sebagai 44 (overflow di parsing!) → selisih = 26 → dianggap usia 44 tahun padahal input 300 yang sebenarnya tidak valid!

Yang lolos adalah input yang setelah truncate ke uint8_t menghasilkan nilai 18–118. Contoh: input 274 → 274 mod 256 = 18 → lolos!

1. Selalu validasi input SEBELUM menyimpan ke tipe terbatas

2. Jangan gunakan unsigned untuk logika "lebih besar dari minimum"

3. Gunakan signed int dengan cek eksplisit: if (umur < 18 || umur > 118)

4. Selalu verifikasi range input di sisi server, bukan hanya client

Soal 3 — Signedness Bug (Signed vs Unsigned Confusion)

Akses memori di luar batas dengan bilangan negatif!

Ini adalah kategori bug yang paling sering menyebabkan buffer overflow! Programmer mencampur signed dan unsigned dalam perbandingan.

char buffer[64] = "Data rahasia: FLAG{secret_data}";
int index; // SIGNED integer!

printf("Masukkan index (0-63): ");
scanf("%d", &index);

// Cek batas: hanya cek batas atas!
if (index < 64) {
  printf("buffer[%d] = %c\n", index, buffer[index]);
} else {
  printf("Index out of range!\n");
}

Pertanyaan: Bagaimana cara membaca data di luar buffer (bahkan sebelum buffer)?

1 / 5
Langkah 1 — Identifikasi celah signedness

Programmer lupa cek batas bawah!

Kode menggunakan int index (SIGNED) tapi hanya cek index < 64. Tidak ada cek index >= 0!

Cek yang ada: if (index < 64) Cek yang kurang: if (index >= 0 && index < 64)

Karena int adalah signed, index bisa berisi nilai negatif. Dan nilai negatif selalu lolos cek index < 64!

index = -1

-1 < 64 → TRUE → akses buffer[-1] → membaca memori SEBELUM buffer!

index = -100

-100 < 64 → TRUE → akses buffer[-100] → baca 100 byte sebelum buffer!

buffer[index] dengan index negatif mengakses memori di luar buffer — bisa membaca stack frame, return address, variabel lokal lain, atau data sensitif!

Apa yang ada di memori sekitar buffer?

Dengan index negatif, kita bisa membaca isi stack sebelum buffer — termasuk variabel lokal, saved registers, bahkan return address!
Langkah 3 — Eksploitasi: baca data sebelum buffer

Trace dengan index = -1, -2, -10

// Input: index = -1
if (-1 < 64) { // TRUE!
  printf("buffer[-1] = %c\n", buffer[-1]);
  // Membaca byte tepat SEBELUM buffer
  // Mungkin isi variabel lokal lain
}

// Input: index = -20
if (-20 < 64) { // TRUE!
  printf("buffer[-20] = %c\n", buffer[-20]);
  // Membaca 20 byte sebelum buffer — bisa stack frame!
}
Dengan mengulang dari -1 sampai -N, attacker bisa melakukan stack read — membaca semua data di stack termasuk canary values, return addresses, dan data sensitif.
Langkah 4 — Kasus nyata: int vs size_t

Bug signedness yang paling umum di dunia nyata

Versi bug yang lebih halus dan sering terjadi:

void copy_data(char* dst, char* src, int len) {
  // len bertipe int (signed)
  memcpy(dst, src, len); // memcpy butuh size_t (unsigned)!
}

// Kalau attacker bisa set len = -1:
// (size_t)(-1) = 0xFFFFFFFF = 4294967295!
// memcpy akan copy 4GB data → CRASH atau arbitrary write!
Ini adalah salah satu bug yang menyebabkan heap overflow! Nilai negatif di-cast ke unsigned menjadi nilai sangat besar.
int len = -1; size_t size = (size_t) len; // implicit cast! // size = 0xFFFFFFFF = 4,294,967,295 bytes!
Bug ini ada di banyak library C klasik. OpenSSL Heartbleed (2014) adalah contoh nyata dari bug membaca memori di luar batas dengan ukuran yang dimanipulasi.

Checklist untuk identifikasi Integer Bug di soal

1. Cari tipe data: uint8, int16, dll — tipe kecil rawan overflow

2. Cari operasi aritmetika: +, -, * pada tipe terbatas

3. Cek apakah ada signed/unsigned mixing dalam perbandingan

4. Cek apakah ada konversi implisit ke tipe lebih besar (int → size_t)

5. Hitung: nilai apa yang menyebabkan wrap ke nilai yang menguntungkan?

Rumus kunci: (hasil_target - nilai_awal + 256) mod 256 = input_yang_dibutuhkan
